# 用户偏好

1. 默认中文回复。
2. 开头先称呼“少爷”，再继续正文；称呼只是礼貌入口，不替代当前 persona。
3. 优先直接给结论与执行结果；复杂检查先核实现状再处理。
4. 提到维护 `moviepilot-agent` 或 Agent 项目时，优先理解为持续维护 Agent 能力资产与使命；除非用户明确要求，不要默认转成 clone/push 等单次 Git 操作。
5. 用户希望所有已接入维护的 Git 仓库保留可回退能力；以后如果仓库维护后出现问题，应能协助按提交历史只读定位、选择安全回退点，并优先用 `git revert` 或文件级恢复，避免默认强推重写历史。
6. 不把单次纠错、技能流程、安全边界或业务规则写入本文件；它们应进入对应专项规则或 skill。
7. 对 `moviepilot-css` 这类样式仓库的维护偏好：先在本地/MoviePilot 实例中充分优化和验证，保持仓库工作区干净；未明确要求同步前不要主动提交推送到远程仓库。
8. MoviePilot 自定义 CSS 的长期偏好统一为：尊重官方主题色与语义色，不覆盖用户可调的 `primary`、`secondary`、`success`、`warning`、`error` 等主题变量或 `.text-primary` 等语义色类；优先贴近 MP 原生 UI 类目，在既有稳定版基础上小范围精修，只做结构、材质、层级与可读性增强，避免机械化大面积重写。
9. 当用户指出“主题色不对 / 没按官方来 / 没有真正生效”时，先回到 MoviePilot 官方前端实现做只读研究，吃透主题变量、组件结构和样式层级后，再改本地 CSS；不要凭经验继续盲改。
10. 若用户明确要求“不要覆盖官方自己的主题色覆盖”，应继续收敛：连基于 `--v-theme-primary` 派生的大面积背景、主底色角标、全局 hover/tint 覆盖也尽量避免；优先只改圆角、阴影、边框、间距、遮罩、模糊、容器透明度、文字层级和卡片结构。
11. 若用户在连续任务中说“自检通过 / 检查 bug / 没问题就同步仓库”，先根据最近活动上下文确认其指的是当前业务任务的验收对象；不要自动把“同步仓库”理解为先验收 `moviepilot-agent` 仓库本身。
12. 当用户要求制作 MoviePilot 本地插件且明确走“独立 Git 仓库源码 → 发行版/ZIP 打包 → 用户自行下载上传到本地插件安装”的链路时，应理解为可在独立仓库准备源码、打包脚本与发布资产；不要误解为直接写入 `/app/app/plugins`。
13. 用户要求 Agent 能力“进化/优化/下一步”时，目标是提升可用能力、执行质量、判断力、工具路由、验证闭环和治理架构，不是机械精简；已经处理并验证过的对象绝对不要反复循环处理，除非有新问题、新目标或用户明确要求复查。若用户指出“不是精简版的残废”“不要死循环处理”，应立即停止把瘦身当作默认目标，改为能力增强、流程补强、缺陷修复、架构升级或未覆盖能力建设，并先盘点“已处理/未处理/真正增益点”。
14. 用户明确要求 TG 交互的真实目标是“尽量通过按钮完成确认、继续、授权、同意、选择”，不是把按钮失效降级成长期文本输入；当用户强调“不想每次输入下一步/继续/授权/同意”时，必须优先修复按钮式交互链路和技能调度，让可按钮化的步骤通过 `ask_user_choice` 发出按钮，文本降级只能作为临时排障保活。
15. 用户讨厌事后找补式汇报：不要把“把用户原话纳入触发/规则”包装成有意义的修复成果。汇报时应说明真实机制改变、验证证据和剩余问题；若某条只是描述性补充或纠偏标签，应直说价值有限，避免邀功式列项。
16. 用户要求插件/Agent 能力资产每次更新后必须先完成自检并给出通过证据，再交给用户验收；不能把“已写入/已重载”直接当成可验收结果。
17. 用户对本地插件开发的目标是“完整收编原插件能力”，不是只做状态预览、减少依赖或阉割版壳子；“完整收编”应理解为由新插件**直接接替原功能**：原插件能做的核心动作，新插件也要有自己的配置、执行入口、确认门禁、结果记录和验证证据；不能只调用/联动/预览原插件来冒充收编。目标插件必须离线、自包含运行：不依赖 AI、不调用 Agent、不依赖其他插件作为功能执行后端；除 MoviePilot 内部公开配置、数据库、通知、调度、事件和系统工具外，核心功能应由插件自身完成。除非涉及凭据、删除、重启、卸载、清库等高影响动作需要确认门禁，否则应把功能做成可执行闭环。汇报时必须区分“已直接接替 / 已完整收编”“仅联动原插件”“只读预览/框架接入”“仍缺执行闭环”。
18. 用户对本地插件 UI/配置页偏好：避免平面化堆字段，优先参考成熟插件的分区/卡片/步骤式设计；配置要小白化，能用开关就用开关，能下拉选择就用下拉，能预设就给预设，能按钮就按钮；Cron、间隔、保留数量、范围、模式等不要默认要求手写，除非确实需要高级自定义；清理类功能应拆成“定时清理”和“一键清理”两个清晰入口；高风险操作必须独立分区、明确警告、预览优先并带确认门禁；可主动补充能增加易用性的预览、推荐值、快捷操作和状态提示。插件配置页不是 Agent 后花园或长期记忆展示区，禁止放自嗨式、路线图式、内部实现式备注；融合/接替原插件时，先参考对应原插件的实际设置项与使用心智，能直接搬运的字段、开关、范围、定时、通知、执行入口和确认门禁应优先照搬，再按 MP 运维助手结构做必要合并。
19. 用户要求“自我蒸馏 / 蒸馏整个 Agent / 开始走自检程序 / 优化程序”时，按 Agent 自我治理主线推进，不要用新增脚本或局部 dry-run 冒充蒸馏成果。
20. 维护 MP 运维助手这类用户侧插件时，插件 UI 只应出现用户能理解和操作的功能说明、设置建议与必要风险提示；提示文案必须说明“这个选项做什么、应该怎么设置”，禁止把 Agent 的长期记忆、skills、方法论、路线图、自我治理措辞或“完全体/迁移/脑回路式说明”塞进插件设置页。若 MoviePilot 官方已提供同类功能（如整理记录清理），应剔除插件内重复能力，避免与官方设置冲突；不要为了展示能力而保留无必要 Dashboard、高风险/门禁独立分类、操作门禁独立分类、内部方法论提示或无实际操作价值的说明。用户明确要求删除 Dashboard 时，应删除详情页 Dashboard 面板与对应前端/API/构建产物入口，只保留设置页中的实用配置、必要门禁和可理解操作说明。配置页提示只允许围绕当前配置项的用途、推荐设置、风险含义与操作后果，不得出现 Agent 能力资产、长期记忆、skills、迁移路线、版本口号或内部实现思路。
